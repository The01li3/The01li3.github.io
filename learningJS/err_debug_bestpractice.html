Errors, Debugging, Best Practice

Errors

Can use a try... catch... finally block.
try will run the code inside it.
catch will run if the try fails with either a system error message or a throw message.
finally gets run regardless of the try/catch result.

In catch(e) the e is the error object, it consists of a name, and a message, display it, console.log it, whatever, its fairly useful to see what went wrong.

Debugging

F12 really helpful, can use it to break at certain points, then do a stepover to see whats causing the error.
Console.log() will also likely get used a lot to see certain variables, and what XYZ happens.
Theres also a debugger; command, that acts the same way as putting in a break point in the debugger.

Strict

"use strict; will make the code run in strict mode.
It can also be used inside a function for block scoping.
Only works if at the begging of the script or function.
Makes it so that the code is less lenient on bad syntax, need to declare variables, less likely to mistype a variable, cannot delete a variable etc.

Styling

Normally use Hungarian notation in camelCase:
vVariable, bBool, cCount, arrArray... etc.

Spaces tend to appear around operators 1 + 1 = 2 and after commas name = ["Alice", "Bob", "Eve"]

W3Schools says to use 2 spaces instead of tabs based off the fact editors can interpret tabs differently... I've always just used tabs.
